    单例模式

    # 单例模式

    保证一个类仅有一个实例，并提供一个访问它的全局访问点。

    构造函数必须是私有的,才能防止外部直接new

    # 场景:

    比如数据库连接池,不会反复创建

    防止线程不安全和提高代码执行效率

    一些全局性质的属性保存,计数器,时间等等

    # 实现方式

    根据实际场景选择是否懒加载以及线程安全

    # 静态类

    运行时就创建好了实例, 通过暴露的静态方法获取或者直接暴露属性 ,

    public class Singleton {
    // 静态字段引用唯一实例:
    private static final Singleton INSTANCE = new Singleton();
    // 通过静态方法返回实例:
    public static Singleton getInstance() {
        return INSTANCE;
    }
    // private构造方法保证外部无法实例化:
    private Singleton() {}}

    # 懒汉模式

    延时加载, 当调用get方法时,才初始化实例

    public class Singleton {
    private static Singleton INSTANCE = null;
    public static Singleton getInstance() {
        //判断实例是否已经创建
        if (INSTANCE == null) {
            //没有则创建
            INSTANCE = new Singleton();
        }
        //返回
        return INSTANCE;
    }
    private Singleton() {
    }
}

    但是上面这种存在线程安全问题, 多线程同时获取时会创建多个实例,那就在方法上加锁

    public synchronized static Singleton getInstance() {
    //判断实例是否已经创建
    if (INSTANCE == null) {
        //没有则创建
        INSTANCE = new Singleton();
    }
    //返回
    return INSTANCE;
}

    方法加锁后会发现,每次获取都会进行加锁, 影响并发性能, 所以可以修改为双重校验锁DCL ( Double Check Lock )

    先判断为null, 则加锁准备初始化, 防止在 判断到获取锁 这段区间其他线程优先执行了初始化,所以获取锁后, 再次判断是否为null

    public static Singleton getInstance() {
    //判断实例是否已经创建
    if (INSTANCE == null) {
        //加锁
        synchronized (Singleton.class) {
            //获取锁后再次判断是否为null
            if (INSTANCE == null) {
                INSTANCE = new Singleton();
            }
        }
    }
    return INSTANCE;
}

    # 枚举类

    利用Java的enum，因为Java保证枚举类的每个枚举都是单例，所以我们只需要编写一个只有一个枚举的类即可：

    public enum World {
    // 唯一枚举:
    INSTANCE;
    private String name = "world";
    public String getName() {
        return this.name;
    }
    public void setName(String name) {
        this.name = name;
    }
}

String name = World.INSTANCE.getName();

    使用枚举实现Singleton还避免了第一种方式实现Singleton的一个潜在问题：即序列化和反序列化会绕过普通类的private构造方法从而创建出多个实例，而枚举类就没有这个问题。

    那我们什么时候应该用Singleton呢？实际上，很多程序，尤其是Web程序，大部分服务类都应该被视作Singleton，如果全部按Singleton的写法写，会非常麻烦，所以，通常是通过约定让框架（例如Spring）来实例化这些类，保证只有一个实例，调用方自觉通过框架获取实例而不是new操作符：

    @Component
// 表示一个单例组件
public class MyService {    ...}

    因此，除非确有必要，否则Singleton模式一般以“约定”为主，不会刻意实现它
