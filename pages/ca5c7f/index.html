springboot框架到2.5.12后redis连接报错

# springboot框架到2.5.12后redis连接报错

根据安全要求, 将java服务的springboot版本升级到2.5.12后, 在河北环境的项目频繁报错redis连接问题

Redis command timed out; nested exception is io.lettuce.core.RedisCommandTimeoutException: Command timed out after 1 minute(s)

如果重复服务便会恢复正常, 网上很多说是lettuce没有心跳检测需要换回jedis但是我看了一下升级前的版本同样也是2.几版本使用的lettuce, 所以没有采用替换使用jedis的方式

参考文档: https://www.cnblogs.com/wingcode/p/14527107.html (opens new window)

目前采用了其中添加心跳机制的方法, 效果还在观测中

lettuce提供了NettyCustomizer进行扩展，熟悉netty的同学，应该听说过netty所提供的心跳机制--IdleStateHandler，结合这两者，就很容易在初始化netty时增加心跳机制：

@Bean
public ClientResources clientResources(){

    NettyCustomizer nettyCustomizer = new NettyCustomizer() {

        @Override
        public void afterChannelInitialized(Channel channel) {
            channel.pipeline().addLast(
                    new IdleStateHandler(readerIdleTimeSeconds, writerIdleTimeSeconds, allIdleTimeSeconds));
            channel.pipeline().addLast(new ChannelDuplexHandler() {
                @Override
                public void userEventTriggered(ChannelHandlerContext ctx, Object evt) throws Exception {
                    if (evt instanceof IdleStateEvent) {
                        ctx.disconnect();
                    }
                }
            });
        }

        @Override
        public void afterBootstrapInitialized(Bootstrap bootstrap) {
        }

    };

    return ClientResources.builder().nettyCustomizer(nettyCustomizer ).build();
}

这里由客户端自己做心跳检测，一旦发现Channel死了，主动关闭ctx.close()，那么ChannelInactived事件一定会被触发了。但是这个方案有个缺点，增加了客户端的压力。
