    # JVM

    # jvm的体系结构

    1. 程序计数器: 当前线程所执行字节码的行号指示器
    2. java虚拟机栈: 描述java方法执行的线程内存模型: 每个方法被执行的时候,java虚拟机都会同步创建一个栈帧用于存储局部变量表、操作数栈、动态链接、方法出口等信息, 每一个方法被调用直至执行完毕的过程, 就对应着一个栈帧在虚拟机栈中从入栈到出栈的过程
    3. 本地方法栈: 和虚拟机栈类似, 不过是为本地方法服务, 虚拟机栈为java方法(字节码)服务
    4. 堆: 存放对象实例
    5. 方法区:存储被虚拟机加载的类型信息、常量、静态变量、即时编译器编译后的代码缓存等数据

    # 类加载过程

    1. 加载
      1. 通过一个类的全限定名获取定义此类的二进制字节流
      2. 将这个字节流所代表的的静态存储结构转化为方法区的运行时数据结构
      3. 在内存中生成一个代表这个类的class对象, 作为方法区这个类的各种数据的访问入口
    2. 链接
      1. 验证
        1. 目的在于确保class文件的字节流中包含的信息符合当前虚拟机要求,保证被加载类的正确性, 不会危害虚拟机自身安全
        2. 主要包括: 文件格式校验, 元数据验证, 字节码验证, 符号引用验证
        3. 简单说就是校验合法性
      2. 准备
        1. 为类变量(静态成员变量)分配内存并且设置该类变量的默认初始值, 即零值
        2. 这里不包含用final修饰的static , 因为final在编译的时候就会分配, 准备阶段会显式初始化
        3. 这里不会为实例变量(非静态成员变量)分配初始化, 类变量会分配在方法区中, 而实例变量是会随着对象一起分配到java堆中
      3. 解析
        1. 将常量池内的符号引用转换为直接引用的过程
        2. 事实上, 解析操作往往会伴随JVM在执行完初始化之后在执行
    3. 初始化
      1. 给类变量量显式赋值

    # 类加载器

    1. 启动类加载器

      用来加载java核心类库, 加载位于/jre/lib目录中的或者被参数 -Xbootclasspath所指定的目录下的核心java类库, 无法被java程序直接引用

    2. 扩展类加载器

      它用来加载java的拓展库, java虚拟机的实现会提供一个扩展库目录, 加载位于/jre/lib/ext目录中的或者java.ext.dirs系统变量所指定的目录下的拓展类库, 该类加载器在此目录里面查找并加载java类

    3. 系统类加载器

      它根据java应用的类路径来加载java类, 一般来说, java应用的类都是由它来加载的, 可以通过ClassLoader.getSystemClassLoader()来获取

    4. 用户自定义类加载器

      通过继承java.lang.ClassLoader类的方式实现

      • 我们需要的类不一定存放在已经设置好的 classPath 下（有系统类加载器 AppClassLoader 加载的路径），对于自定义路径中的 class 类文件的加载，我们需要自己的 ClassLoader。
      • 有时我们不一定是从类文件中读取类，可能是从网络的输入流中读取类，这就需要做一些加密和解密操作，这就需要自己实现加载类的逻辑，当然其他的特殊处理也同样适用。
      • 可以定义类的实现机制，实现类的热部署,如 OSGi 中的 bundle 模块就是通过实现自己的 ClassLoader 实现的。

    # 双亲委派机制

    1. 工作原理
      1. 如果一个类加载器收到了类加载请求, 他并不会自己先加载, 而是把这个请求委托给父类的加载器去执行
      2. 如果父类存在父类加载器, 则进一步向上委托,依次递归, 请求最终将到达顶层的启动类加载器
      3. 如果父类加载器可以完成类加载任务, 就成功返回,
      4. 如果父类无法完成则向下子类尝试加载
    2. 目的
      1. 避免类的重复加载
      2. 保护程序安全, 防止核心api被随机篡改

    # 常见垃圾回收算法

    1. 引用计数: 引用依次计数+1, 如果计数为0表示没引用可清除 缺点: 每次对象赋值都要维护引用计数器, 且计数器本身也有消耗; 较难处理循环引用 JVM的实现一般不采用

    2. 复制

      一般用于新生代(又细分Eden区、From Survivor区和 To Survivor区) 比例8:1:1

      过程: 复制→清空→互换

      将eden和FormSurvivor中还存活的对象拷贝到ToSurvivor中, 同时这些对象年龄+1

      清空eden和FormSurvivor中的对象

      FormSurvivor和ToSurvivor身份互换, 谁空谁是to; 如果两个区交换15次后,还存活进入老年代

    3. 标记清除

      一般用于老生代, 分为标记和清除两个阶段, 先标记出要回收的对象, 然后同一回收 缺点: 内存碎片过多, 不连续

    4. 标记整理

      一般用于老生代, 分为标记和压缩两步, 先标记,然后再次扫描, 并向一端滑动存活数据, 没有内存碎片

      缺点: 需要移动对象

    # JVM垃圾回收时如何确定对象? 什么是GC Roots?

    垃圾: 简单来说就是内存中已经不再被使用的空间就是垃圾

    怎么确定可以被回收:

    1. 引用计数法: 维护引用计数, 已经不使用了
    2. 枚举根节点做可达性分析(跟搜索路径)

    GC Roots 就是一组必须活跃的引用

    通过GC Roots 为起始点, 向下搜索, 如果一个对象到起始点没有任何引用链相连时, 则说明该对象不可用.

    可做GC Roots的对象:

    1. 虚拟机栈(栈帧中的局部变量区, 也叫作局部变量表)中引用的对象
    2. 方法区中的类常量属性引用的对象
    3. 方法区中常量引用的对象
    4. 本地方法区中JNI(Native方法)引用的对象

    # JVM参数类型

    1. 标配参数: 各个版本一直没变
      1. -version
      2. -help
      3. -showversion
    2. X参数(了解)
      1. -Xint 解释运行
      2. -Xcomp 第一次使用就编译成本地代码
      3. -Xmixed 混合模式
    3. XX参数(重点)

      1. 布尔类型

        语句格式: -XX:+/- 某个属性 + 表示开启 -表示关闭 查看某属性是否开启

        1.是否打印GC收集细节

        -XX:-PrintGCDetails 关闭

        -XX:+PrintGCDetails 开启

        在运行中查看是否开启 : jinfo -flag PrintGCDetails 进程号

        1. 是否使用串行垃圾回收器

          -XX:-UserSerialGC

          -XX:+UserSerialGC

      2. KV设值类型

        语句格式: -XX属性key=属性值value

        1. -XX:MetaspaceSize=128m 设置元空间初始值
        2. -XX:MaxTenuringThreshold=15 设置年轻代到年老区的岁数
    4. jps/jinfo
      1. jps -l 查看当前进程 类似ps
      2. jinfo -flags 查看所有属性
      3. jinfo -flag 属性名 查看单个属性

    # -Xms和-Xmx

    -Xms 等价于 -XX:InitialHeapSize 初始化堆内存

    -Xmx 等价于 -XX:MaxHeapSize 最大堆内存

    # 查看初始默认值

    1. -XX:+PrintFlagsInitial 主要查看初始默认值
      1. java -XX:+PrintFlagsInitial 可以添加在运行参数上也可以直接黑窗口用 查询出来的列表如果等号前有冒号代表修改过
    2. -XX:+PrintFlagsFinal
    3. -XX:+PrintCommandLineFlags

    # JVM常用配置参数

    1. -Xms

      1. 初始化堆内存大小, 默认为物理内存的1/64
      2. 等价 -XX:InitialHeapSize

    2. -Xmx

      1. 最大分配堆内存, 默认为物理内存的1/4
      2. 等价 -XX:MaxHeapSize

    3. -Xss

      1. 设置单个线程栈的大小 一般默认512k-1024k
      2. 等价 -XX:ThreadStackSize

    4. -Xmm

      1. 设置年轻代的大小

    5. -XX:MetaspaceSize

      1. 元空间大小 : 元空间使用本地内存, 默认情况下,元空间的大小仅受本地内存限制

    6. -XX+PrintGCDetails

      1. 输出详细GC收集日志信息

    7. -XX:SurvivorRation

      1. 设置新生代中eden和s0/s1空间的比例
      2. 默认是8:1:1
      3. 例: -XX:SurvivorRatio=4 设置eden的比例 也就是4:1:1

    8. -XX:NewRatio

      1. 配置新生代和老年代在堆结构的占比
      2. 默认是1:2
      3. 例: -XX:NewRatio=4 设置的老年代的占比 新生代1 老年代4

    9. -XX:MaxTenuringThreshold

      1. 新生代经过几次GC后能到老年代
      2. 默认15次 设置范围0-15

    10. 典型配置案例

      -Xms128m -Xmx4096m -Xss1024k -XX:MetaspaceSize=512m -XX:+PrintCommandLineFlags -XX:+PrintGCDetails -XX:+UserSerialGC(默认是并行, 串行在服务器环境不用)

    # 引用

    1. 强引用
      1. 平时我们=就是强引用, 不会被回收
      2. 例 : Object o1 = new Object(); Object o2 = o1; o1=null; System.gc(); 这是o1回收了, o2不会回收
    2. 软引用
      1. 内存够用不回收, 内存不够用才回收
      2. 用java.lang.ref.SoftReference实现
      3. 例: SoftReference<Object> softReference = new SoftReference<>(o1); 这就是软引用,获取方法→softReference.get()
    3. 弱引用\
      1. 只要进行垃圾清理, 都会回收
      2. 用java.lang.ref.WeakReference类实现
      3. WeakHashMap
    4. 虚引用
      1. 需要用java.lang.ref.PhantomReference类实现

    # OOM

    1. 常见的
      1. java.lang.StackOverflowError
        1. 栈溢出 比如递归深度过深
      2. java.lang.OutOfMemoryError: java heap space
        1. 堆内存溢出
      3. java.lang.OutOfMemoryError: GC overhead limit exceeded
        1. GC回收时间过长, 超过98%的时间来GC并且回收了不到2%的堆内存
        2. GC频繁执行,但是效果甚微,造成了循环GC
      4. java.lang.OutOfMemoryError: Direct buffer memory
        1. 直接内存溢出
        2. 本地内存, 不属于GC范围, 本地内存用光了就会崩溃
        3. 一般出现在NIO程序中, NIO经常使用BateBuffer来读取和写入
        4. ByteBuffer.allocateDirect() 会分配本地内存
      5. java.lang.OutOfMemoryError: unable to create new native thread
        1. 导致原因:
          1. 你的应用创建了太多线程, 一个应用进程创建多个线程, 超过系统承载极限
          2. 你的服务器不允许你的应用创建这么多线程, linux系统默认单个进程可以创建的线程数是1024个
        2. 解决办法:
          1. 想办法降低应用创建线程的数量
          2. 修改linux的服务器配置, 扩大默认限制
            1. ulimit -u 查看允许的线程数
            2. vim /etc/security/limits.d/90-nproc.conf 修改这个文件
      6. java.lang.OutOfMemoryError: Metaspace

    # 垃圾回收器

    1. Serial 串行垃圾回收器
      1. 为单线程环境设计且只使用一个线程进行垃圾回收,会暂停所有的用户线程. 不适合服务器环境
    2. Parallel 并行垃圾回收器
      1. 多个垃圾收集线程并行工作, 此时用户线程是暂停的, 适用于科学计算/大数据处理首台处理等弱交互场景
    3. CMS 并发垃圾回收器
      1. 用户线程和垃圾收集线程同时执行(不一定并行,可能交替进行), 适用于对响应时间有要求的场景
    4. G1垃圾回收器
      1. 将堆内存分割为不同的区域然后并发的对其进行垃圾回收

    # 如何选择垃圾收集器

    1. 单cpu或小内存, 单机程序

      -XX:+UseSerialGC

    2. 多cpu, 需要最大吞吐量, 如后台计算型应用

      -XX:+UseParallelGC 或者 -XX:+UseParallelOldGC

    3. 多cpu, 追求低停顿时间, 需要快速响应如互联网应用

      -XX:+UseConcarkSweepGC -XX:+ParNewGC

    1. G1没有碎片
    2. 可以精确的控制停顿, G1将整个堆分为多个固定大小的区域, 每次根据允许停顿的时间去收集垃圾最多的区

    # 对象创建的步骤

    1. 加载类元信息
    2. 为对象分配内存
    3. 处理并发问题
    4. 属性的默认初始化(零值初始化)
    5. 设置对象头信息
    6. 属性的显式初始化、代码块中初始化、构造器中初始化

    # 对象内存布局

    1. 对象头

      1. 运行时元数据
        1. 哈希值
        2. GC分代年龄
        3. 锁状态标志
        4. 线程持有的锁
        5. 偏向锁id
        6. 偏向时间戳
      2. 类型指针 : 指向类元数据, 确定该对象所属的类型
      3. 如果是数组,还要记录数组长度

    2. 实例数据

      是对象真正存储的有效信息, 包含程序代码中定义的各个类型的字段(包含自身的父类继承的)

      1. 相同宽度的字段总是被分配在一起
      2. 父类中定义的变量会在子类之前
      3. 如果CompactFields参数为true(默认为true), 子类的窄变量可能插入到父类变量的空隙

    3. 对齐填充: 不是必须的, 也没有特别含义, 起到占位符的作用
